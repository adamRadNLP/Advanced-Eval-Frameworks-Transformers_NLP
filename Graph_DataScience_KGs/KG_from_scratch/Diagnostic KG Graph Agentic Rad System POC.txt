# Diagnostic Knowledge Graph Agentic Radiology System - Research & POC Plan

## Context

The goal is to build a **human-in-the-loop agentic veterinary radiology reporting system** that:
1. Extracts diagnostic knowledge graphs (KGs) from a corpus of **canine radiograph reports**
2. Aligns available information (from AI classifiers + radiologist input) to highest-matching graph paths
3. Works with the **veterinary radiologist** to connect missing edges and generate a complete report
4. Integrates existing **ViT+SigLIP condition/findings classifiers** and a hanging rules model (anatomy)

**Domain**: Canine radiographs (X-ray studies, not CT)
**Existing models**: ViT-based image encoder fine-tuned on canine images/reports with SigLIP loss + CXRBert text models
This is for a **week-long hackathon POC**.

| Column | Type | Description |

| `weight_lbs` | float | Weight in pounds (note: some entries are 0.0) |

| `report_attention_mask` | list[int] | Attention mask for the tokenized report (1 = real token, 0 = padding) |
| `report` | str | **Detokenized full radiology report** (added locally via CXRBert BertTokenizer from `microsoft/BiomedVLP-CXR-BERT-specialized`) |

### Report Structure
The `report` column contains full radiology reports with a consistent structure:
1. **Patient information**: species, breed, age (months), gender, weight
2. **History**: clinical notes / presenting complaint
3. **Findings**: detailed radiographic observations by body region, organized by view (lateral, VD/DV)
4. **Conclusions/Impression**: differential diagnoses ranked by likelihood, recommendations

### Notes
- Reports are tokenized with the [microsoft/BiomedVLP-CXR-BERT-specialized](https://huggingface.co/microsoft/BiomedVLP-CXR-BERT-specialized) tokenizer (BERT WordPiece vocabulary)
- The `report` column was reconstructed by decoding `report_input_ids` masked by `report_attention_mask` with `skip_special_tokens=True`
- Some `weight_lbs` values are 0.0 (missing data)
- Each case can have multiple images (list of URLs/asset IDs)
- Reports vary in length and detail; some are brief pre-anesthetic workups, others are comprehensive multi-region studies

## Part 1: Source Review & Analysis

### 1.1 KARGEN (Knowledge Enhanced Automated Radiology Report Generation)
**Paper**: [arxiv.org/html/2409.05370v1](https://arxiv.org/html/2409.05370v1)

**What it does**: Integrates LLMs (LLaMA2-7B) with a disease-specific knowledge graph for automated radiology report generation using a 3-layer GCN over 14 CheXpert disease entities.

**Architecture**: Swin Transformer (visual features) -> GCN over disease KG -> Feature fusion (element-wise or MoE-style) -> LLaMA2-7B report generation

**Key insight for our system**: Their KG is small (14 nodes) and static. Our approach should build **richer, dynamic KGs** from report corpora. However, their fusion approach (gating visual + graph features) is relevant for combining CLIP classifier outputs with KG paths.

**Limitations**: Fixed 14-disease graph; no human-in-the-loop; no iterative refinement.

---

### 1.2 RadGraph / RadGraph-XL
**Dataset**: [physionet.org/content/radgraph/1.0.0](https://physionet.org/content/radgraph/1.0.0/)
**RadGraph-XL**: [aclanthology.org/2024.findings-acl.765](https://aclanthology.org/2024.findings-acl.765/)
**Python package**: [github.com/Stanford-AIMI/radgraph](https://github.com/Stanford-AIMI/radgraph) (`pip install radgraph`)

**What it provides**:
- **Entity types**: ANAT-DP (anatomy present), OBS-DP (observation present), OBS-U (uncertain), OBS-DA (absent)
- **Relation types**: `suggestive_of`, `located_at`, `modify`
- **RadGraph-XL** (2024): 2,300 reports, 410K+ entities/relations, covers chest CT, abdomen/pelvis CT, brain MR, chest X-ray
- **Trained models**: DYGIE++ on PubMedBERT, F1=0.82 (MIMIC-CXR)
- **Python API**: `RadGraph(model_type="modern-radgraph-xl")` - extract entities/relations from any report text

**Critical for our system**: RadGraph is the **foundation for KG extraction**. The `radgraph` Python package can directly extract entity-relation graphs from our report corpus. The entity/relation schema is well-suited for building diagnostic KGs. RadGraph-XL's multi-modality coverage (CT, MR, CXR) is a major advantage.

### 1.3 Semantic Entity Resolution (Graphlet.ai)
**Source**: [blog.graphlet.ai](https://blog.graphlet.ai/the-rise-of-semantic-entity-resolution-45c48d5eb00a) (Medium paywall)

**Concept**: Merging duplicate/variant entities in knowledge graphs using:
- Embedding similarity (cosine threshold)
- LLM-based disambiguation
- UMLS normalization for medical terms
- Graph structural features

**Relevance**: After extracting entities from many reports, we'll have duplicates and variants (e.g., "cardiomegaly" vs "enlarged heart" vs "cardiac enlargement"). Entity resolution is **essential** for building a clean, unified KG. ReXKG uses 0.95 embedding similarity + ScispaCy/UMLS normalization - a practical approach we can adopt.

### 1.4 Google RadExtract / LangExtract
**Demo**: [huggingface.co/spaces/google/radextract](https://huggingface.co/spaces/google/radextract)
**Library**: [github.com/google/langextract](https://github.com/google/langextract) (`pip install langextract`)
**Docs**: [developers.google.com/health-ai-developer-foundations/libraries/langextract](https://developers.google.com/health-ai-developer-foundations/libraries/langextract)

**What it does**: LLM-powered structured extraction from unstructured text with precise source grounding. Uses Gemini 2.5 Flash/Pro. Few-shot learning, no fine-tuning needed. Returns structured JSON with character-level source attribution.

**Key features**:
- Schema-driven extraction (define what to extract)
- Source grounding (every extraction maps to exact text location)
- Interactive HTML visualization for verification
- Handles long documents via chunking + parallel processing
- Supports Gemini, OpenAI, Ollama (local models)

**Relevance**: Could be used as an **alternative or complement to RadGraph** for KG extraction. Particularly useful for extracting textbook knowledge into structured form. The source grounding is valuable for the human-in-the-loop verification step.

### 1.5 Anthropic's Building Effective Agents Guide
**Source**: [anthropic.com/engineering/building-effective-agents](https://www.anthropic.com/engineering/building-effective-agents)

**Key patterns relevant to our system**:

1. **Orchestrator-Workers**: Central LLM breaks tasks into subtasks for specialized workers. Maps to our system where an orchestrator manages the KG matching + radiologist interaction while delegating to CLIP classifiers, KG search, etc.

2. **Evaluator-Optimizer**: Iterative refinement loop. Maps to the radiologist reviewing generated report sections and providing feedback.

3. **Routing**: Classify input and direct to specialized handler. Maps to hanging rules model routing images to anatomy-specific processing paths.

4. **Human-in-the-loop**: Agents should pause for human feedback at checkpoints. Core to our design.

**Critical guidance**: "Start simple. Add complexity only when it demonstrably improves outcomes." For hackathon: start with prompt chaining, not a full multi-agent system.

## Part 2: Additional SOTA Methods Discovered

### 2.1 ReXKG (Rajpurkar Lab, 2024)
**Paper**: [arxiv.org/abs/2408.14397](https://arxiv.org/abs/2408.14397)
**Code**: [github.com/rajpurkarlab/ReXKG](https://github.com/rajpurkarlab/ReXKG)

**What it does**: Full pipeline for constructing radiology KGs from reports:
1. GPT-4 annotation -> PURE NER/RE training
2. Entity extraction (anatomy, disorder, concept, device, procedure, size)
3. Node construction (ScispaCy + UMLS normalization, 0.95 embedding similarity merging)
4. Edge construction (triplet dedup, frequency filtering >= 5 occurrences)

**3 evaluation metrics**: KG-NSC (node similarity), KG-AMS (edge distribution), KG-SCS (subgraph coverage)

**Key finding**: Generalist models achieve ~80% entity coverage but miss complex relationships. The top 10% most frequent subgraphs capture the most important diagnostic patterns.

**Relevance**: ReXKG provides a **complete, open-source pipeline** for our KG construction step. Their frequency-based subgraph importance aligns perfectly with our concept of matching to "highest matching graph paths."

### 2.2 R2GenKG (2025)
**Paper**: [arxiv.org/html/2508.03426](https://arxiv.org/html/2508.03426)

**What it does**: Builds M3KG (Multi-Modal Medical KG) with 2,477 entities, 37K+ triples, and disease-aware vision tokens. Uses multi-scale graph representations (coarse to fine) with R-GCN encoding and bidirectional cross-attention between vision and KG features.

**Key innovation**: Multi-scale graph views - coarse subgraphs filter noise, fine-grained graphs retain detail. Achieves SOTA on IU X-Ray (BLEU-4: 0.181, CIDEr: 0.701).

**Relevance**: The multi-scale graph concept is valuable for our path matching - start with coarse anatomy-level matching, then drill down to finding-level detail.

### 2.3 BiomedCLIP
**Paper**: [arxiv.org/abs/2303.00915](https://arxiv.org/abs/2303.00915)

Pretrained on 15M biomedical image-text pairs from PMC. Outperforms radiology-specific models (BioViL) on tasks like RSNA pneumonia detection. Available on HuggingFace. Directly applicable as the backbone for condition and findings classifiers.

### 2.6 Veterinary-Specific Research

**Companion Animal Disease KG** ([arxiv.org/abs/2309.03219](https://arxiv.org/abs/2309.03219)):
- LiteralKG model builds KGs from veterinary EMRs across animal hospitals
- Uses gate networks to fuse entity types with literal attributes into unified representations
- Self-supervised pretraining on graph structure for downstream diagnostic tasks
- Outperforms SOTA on link prediction (disease relationship prediction)
- Code is publicly available

**ACVR/ECVDI Consensus on Imaging Reports** ([PMC11649853](https://pmc.ncbi.nlm.nih.gov/articles/PMC11649853/)):
- Canonical vet radiology report structure: **Header -> Findings -> Impression -> Signature**
- Findings section: "all abnormal observations and pertinent normal findings" with standardized technical language
- Impression section: clinical context, differential diagnoses, recommendations
- Diagnosis format: **severity + acuity + distribution** (e.g., "Severe, acute gastric dilatation-volvulus")
- Emphasizes definitive diagnoses when confident, prioritized differentials when uncertain
- **This defines our KG's target output schema**

**SignalRAY/SignalPET (Commercial Reference)** ([Frontiers 2025](https://www.frontiersin.org/journals/veterinary-science/articles/10.3389/fvets.2025.1502790/full)):
- 38 deep learning architectures, trained on 2.5M+ canine/feline radiographs from 3,000+ practices
- Provides binary normal/abnormal classification per finding
- Matches best radiologist accuracy, more specific but less sensitive
- Cannot provide differential diagnoses or clinical context - **exactly the gap our system fills**

**VetSCT (Veterinary SNOMED-CT Extension)** ([vtsl.vetmed.vt.edu](https://vtsl.vetmed.vt.edu/extension/)):
- Maintained by Virginia Tech Center for Veterinary Terminology and Standards
- Extends SNOMED-CT with animal-specific content
- Browsable terminology database (Feb 2025 release)
- Requires signup and approval process -- **deferred to post-hackathon**
- For POC, CXRBert embeddings + LLM-based normalization provide equivalent entity resolution without external dependencies

| Tool | Transferability | Adaptation Needed |

## Part 3: Agentic Framework Comparison

### Option A: Claude Code with Skills/MCPs
**Pros**:
- Fastest to prototype (you already have the environment)
- MCP servers for Neo4j, knowledge graphs already exist
- Skills can encapsulate complex workflows
- Human-in-the-loop is native (the radiologist IS the user)
- Follows Anthropic's "start simple" guidance
- Direct access to Claude's reasoning for graph path matching

**Cons**:
- Less structured multi-agent orchestration
- Harder to integrate custom CLIP models (would need MCP wrapper)
- Single-threaded conversation model

### Option B: CrewAI / Bespoke Framework
**Pros**:
- Purpose-built multi-agent coordination
- Role-based agents (KG agent, classifier agent, report writer agent)
- Better for production deployment
- 5.76x faster execution than LangGraph in benchmarks

**Cons**:
- More setup time (significant for a hackathon)
- Need to build the human-in-the-loop interaction from scratch
- Framework overhead for what may be a sequential workflow
- Overkill if the core logic is actually a pipeline, not truly multi-agent

### Recommendation for Hackathon: **Hybrid Approach**
Use Claude Code as the orchestrator and human interface, with MCP servers wrapping the KG database and CLIP classifiers. This gives you:
- Natural human-in-the-loop (radiologist talks to Claude)
- MCP-based tool access to KG queries, CLIP inference, report templates
- Fast prototyping with the option to extract into a standalone system later


### Workflow Concept

1. **Study arrives** -> Hanging rules model identifies anatomy/body region
2. **Classifiers run** -> ViT+SigLIP models produce condition + finding predictions with confidence scores
3. **Graph matching** -> Predictions mapped to KG nodes; find highest-matching diagnostic subgraph paths
4. **Gap analysis** -> Agent identifies missing edges (e.g., "model detects cardiomegaly but no vertebral heart score measurement")
5. **Radiologist interaction** -> Agent presents matched paths + gaps to vet radiologist, asks targeted questions
6. **Report generation** -> Complete graph path converted to ACVR-format report (Findings + Impression)

- **ViT+SigLIP condition + findings classifiers** (trained, ready to integrate)
- **CXRBert text encoder** (trained on canine reports)
- **Hanging rules model** (anatomy identification, trained)

#### Phase 1 (Days 1-2): Knowledge Graph Construction from Canine Reports

**1a. Define entity/relation schema** (adapted from RadGraph for canine anatomy + ACVR report format):

```
Entity Types:
  - ANATOMY       (body region/structure, e.g., "thorax", "left stifle", "caudal vena cava")
  - FINDING       (radiographic observation, e.g., "increased opacity", "periosteal reaction")
  - CONDITION     (diagnosis/disease, e.g., "cardiomegaly", "hip dysplasia", "GDV")
  - MODIFIER      (severity/acuity/distribution per ACVR, e.g., "severe", "chronic", "diffuse")

Relation Types:
  - located_at     (FINDING -> ANATOMY)
  - suggestive_of  (FINDING -> CONDITION, or FINDING -> FINDING)
  - modify         (MODIFIER -> FINDING/CONDITION)
  - differential_for (CONDITION -> CONDITION, ordered by likelihood)
  - absent         (negated FINDING at ANATOMY)
```

**1b. Extract entities/relations from report corpus** (mixed format - some structured, some free-text):
- First: parse structured reports to separate Findings/Impression sections where available
- For free-text reports: use LLM to identify section boundaries
- ~~Option A (recommended): Use **Claude API** with structured output schema + few-shot examples from your reports~~
- ~~Option B: Use **LangExtract** (Gemini-powered, source grounding, batch processing)~~
- **IMPLEMENTED**: Using **Gemini 2.5 Flash** on Vertex AI with Pydantic JSON schema enforcement (see `extract.py`)
- Sample 500-1000 reports for initial extraction, validate, then scale to full 10K+
- Run extraction in batches with parallel processing

**1c. Entity resolution (CXRBert + LLM approach, no external ontology required)**:
1. Embed all extracted entities using **CXRBert CLIP text encoder v4** (`overlook-dev.radiology_classifier.clip_rad_dino/4`), 768-dim embeddings — v4 projection head fine-tuned on full anatomy radiographs (v7 is abdomen-only)
2. Cluster entities by cosine similarity using **per-type thresholds** (ANATOMY=0.93, FINDING=0.95, CONDITION=0.93, MODIFIER=0.97) — higher thresholds for types where CXRBert poorly discriminates short generic words. Agglomerative clustering with complete linkage (all pairs within cluster must meet threshold).
3. For each multi-member cluster, use **Gemini 2.5 Flash** to select the **canonical name** (most specific/standard term). Chosen over Gemini 3 Flash Preview after comparison: 2.5 Flash is faster, more conservative (picks existing cluster members 83% of the time), and more reliable. See `output/llm_canonical_comparison.md`.
4. Merge cluster members into single KG nodes with aliases preserved

This approach is stronger than ontology-based normalization for our domain because CXRBert understands the actual language our radiologists use. VetSCT integration is deferred to post-hackathon (see Future Work).

**1c-augmented. LLM-augmented entity resolution (A+B hybrid)**:

Baseline entity resolution (1c above) achieves precision=86.4%, recall=63.7%, F1=0.734 on the 206-pair evaluation set. The embedding-based approach struggles with short generic words and misses abbreviation/synonym pairs that fall below the cosine threshold. Augment with two LLM passes:

**Pass A — Cluster Validation (precision fix)**:
- For each multi-member cluster from step 1c.2, send to LLM: "Are all of these terms truly synonymous in veterinary radiology? If not, which should be split out?"
- Split clusters where the LLM identifies false merges
- Cost: ~1,696 LLM calls (~$0.50-1.00), ~15 min
- Target: precision 86.4% → **≥95%**

**Pass B — Near-Threshold Pair Rescue (recall fix)**:
- Identify entity pairs that fall just below the clustering threshold (e.g., cosine similarity 0.80–0.93 for ANATOMY/CONDITION, adjusted per type)
- Send borderline pairs to LLM: "Are these two veterinary radiology terms synonymous?"
- Merge confirmed pairs into existing or new clusters
- Catches: abbreviations (GDV ↔ gastric dilatation-volvulus), lay-vs-medical terms, multi-word rephrasings
- Cost: ~500-2,000 LLM calls (~$0.50-2.00), ~15-20 min
- Target: recall 63.7% → **≥80%**

Combined target: **precision ≥95%, recall ≥80%, F1 ≥0.87**. Evaluate against the 206-pair benchmark set.

Precision is prioritized over recall because false merges create incorrect diagnostic paths in the KG (clinically dangerous), while missed merges only fragment the graph (compensated at query time by soft node matching — see 2e).

**Pass C — Abbreviation / Acronym Lookup (TODO)**:
- Projection-based evaluation reveals that short medical abbreviations (CCL, GDV, IVDD, CVC) are unsolvable by CXRBert CLIP — they embed into generic regions (e.g., "CCL" → "tall" at sim=0.991, "GDV" → "vhs" at sim=0.890, "IVDD" → "liver atrophy" at sim=0.637)
- Longer/more distinctive abbreviations fare better: CHF → "congestive heart failure" (correct), DJD → "degenerative joint disease" (correct)
- **Solution**: curate or LLM-generate a veterinary radiology abbreviation lookup table mapping acronyms to their full expansions, and apply it as a pre-processing step before embedding
- Alternatively, add a Pass C that specifically targets short uppercase tokens (≤5 chars, all caps) and asks the LLM to expand them before resolution
- Expected to recover most of the remaining FN pairs in the projected evaluation (8 TP → potentially 12-15+ TP)

**1d. Build and analyze KG**:
- Construct in **Neo4j** (graph database with Cypher queries, visualization, MCP integration)
- Run Neo4j via Docker locally
- Filter edges by frequency (>= 5 occurrences, per ReXKG)
- Compute subgraph frequency and importance scores
- Identify **top diagnostic paths**: anatomy -> finding(s) -> condition with modifiers
- Also maintain NetworkX representation for algorithmic analysis (path finding, subgraph matching)

**2e. Graph path matching algorithm (with soft node matching)**:
- Input: classifier predictions (conditions + findings + anatomy)
- **Soft node matching**: Map each prediction to top-K candidate KG nodes by CXRBert embedding similarity (not just nearest-1). Weight candidates by match quality score `s_match = cosine_sim(query, node)`. This compensates for residual entity resolution fragmentation — if "liver enlargement" and "hepatomegaly" are separate nodes, a query for "enlarged liver" activates both with appropriate weights.
- **Neighborhood expansion**: For each candidate node, retrieve its local subgraph (1-2 hop neighborhood). Union overlapping neighborhoods weighted by match score.
- **Path finding**: Find all diagnostic paths (ANATOMY ← FINDING → CONDITION) that pass through any activated node.
- **Composite path scoring**: Score each path by:
  - (a) `s_match`: node match quality (how well did the query term match the path's nodes)
  - (b) `s_freq`: path frequency in KG (edge frequency, importance_score)
  - (c) `s_complete`: path completeness (fraction of classifier predictions covered by this path)
  - Combined: `score = s_match^α × s_freq^β × s_complete^γ` (tune weights on held-out cases)
- Return top-K paths with gap analysis (missing edges)
- See research notes on soft matching: spreading activation (Collins & Loftus 1975), PersonalizedPageRank for local subgraph scoring, and KGQA soft entity linking (ELQ, BLINK)

#### Phase 3 (Days 5-6): Human-in-the-Loop Workflow

**The core interaction loop** (Anthropic's Evaluator-Optimizer pattern):

```
1. Agent: "I analyzed the thoracic radiograph. Here's what I found:"
   - Lists classifier predictions with confidence
   - Shows top 3 matching diagnostic paths from KG
   - Highlights missing edges: "I see increased bronchointerstitial opacity
     but need your assessment of distribution pattern (focal vs diffuse)"

2. Radiologist: Confirms/rejects findings, provides missing info

3. Agent: Updates graph, re-matches, presents refined paths
   - "With diffuse pattern confirmed, top differential path is:
     diffuse bronchointerstitial opacity -> suggestive_of -> chronic bronchitis
     (seen in 847 reports) vs allergic bronchitis (312 reports)"

4. Radiologist: Selects/modifies diagnosis

5. Agent: Generates ACVR-format report draft from completed path
   - Findings section: All observations + pertinent normals
   - Impression section: Diagnosis (severity + acuity + distribution) + differentials + recommendations

6. Radiologist: Reviews, edits, approves
```

- **Abbreviation lookup table (Pass C)**: CXRBert CLIP cannot resolve short medical abbreviations (CCL, GDV, IVDD, CVC) — they embed into generic regions. Curate a vet radiology acronym → expansion table (or LLM-generate one) and apply as pre-processing before embedding. See 1c-augmented Pass C notes.
- **VetSCT Integration**: Map KG entities to Veterinary SNOMED-CT codes for interoperability and standardized terminology. Requires signup/approval at [vtsl.vetmed.vt.edu](https://vtsl.vetmed.vt.edu/extension/).
- **Production agentic framework**: Evaluate migration from Claude Code + MCP to CrewAI or similar for multi-user deployment
- **Multi-species support**: Extend KG schema and classifiers beyond canine to feline and other species
- **Continuous KG enrichment**: Pipeline to ingest new reports and update the KG incrementally
- **Evaluation framework**: Formal radiologist evaluation study comparing report quality with/without the co-pilot
